🔍 QueryDSL 동적 쿼리

Product, Order, Post 도메인에서 QueryDSL의 BooleanBuilder와 PathBuilder를 활용하여 동적 검색/필터링/정렬을 구현합니다.

ProductRepositoryImpl — 동적 검색 구현 예시

@RequiredArgsConstructor
public class ProductRepositoryImpl implements ProductRepositoryCustom {

    private final JPAQueryFactory queryFactory;

    @Override
    public Page<Product> search(ProductSearchCondition cond, Pageable pageable) {
        QProduct product = QProduct.product;

        // 1. 동적 WHERE 조건 구성 (BooleanBuilder)
        BooleanBuilder where = new BooleanBuilder();
        if (cond.category() != null)
            where.and(product.category.eq(cond.category()));
        if (cond.name() != null)
            where.and(product.name.containsIgnoreCase(cond.name()));
        if (cond.minPrice() != null)
            where.and(product.price.goe(cond.minPrice()));
        if (cond.maxPrice() != null)
            where.and(product.price.loe(cond.maxPrice()));
        if (Boolean.TRUE.equals(cond.inStock()))
            where.and(product.stock.gt(0L));

        // 2. 동적 정렬 (PathBuilder로 문자열 → QueryDSL OrderSpecifier 변환)
        List<OrderSpecifier<?>> orders = new ArrayList<>();
        PathBuilder<Product> path = new PathBuilder<>(Product.class, "product");
        for (Sort.Order sortOrder : pageable.getSort()) {
            orders.add(new OrderSpecifier<>(
                sortOrder.isAscending() ? Order.ASC : Order.DESC,
                path.getComparable(sortOrder.getProperty(), Comparable.class)));
        }

        // 3. 쿼리 실행
        List<Product> content = queryFactory.selectFrom(product)
            .where(where)
            .orderBy(orders.toArray(new OrderSpecifier[0]))
            .offset(pageable.getOffset())
            .limit(pageable.getPageSize())
            .fetch();

        return new PageImpl<>(content, pageable, total);
    }
}
검색 조건 Record: ProductSearchCondition(category, name, minPrice, maxPrice, inStock)는 Java Record로 정의되어 불변이며 null 가능한 필드로 "있으면 적용, 없으면 무시" 패턴을 구현합니다.

🔒 트랜잭션 & 비즈니스 로직

OrderService.create() — 주문 생성 + 재고 차감

부하 테스트의 핵심 대상인 주문 생성 로직입니다. 하나의 트랜잭션 안에서 사용자 검증, 재고 차감, 주문 생성이 원자적으로 처리됩니다.

@Transactional
public Order create(Long userId, String shippingAddress,
                    Map<Long, Long> productQuantities) {
    // 1. 사용자 존재 확인
    User user = userRepository.findById(userId)
        .orElseThrow(() -> new ResourceNotFoundException("User", userId));

    // 2. 상품 조회 + 재고 차감 (실패 시 BusinessException → 400)
    List<Product> products = productRepository.findAllById(productQuantities.keySet());
    for (Product product : products) {
        Long qty = productQuantities.get(product.getId());
        if (!product.decreaseStock(qty)) {
            throw new BusinessException(
                "Insufficient stock for product: " + product.getName());
        }
        totalPrice = totalPrice.add(product.getPrice().multiply(BigDecimal.valueOf(qty)));
    }

    // 3. 주문 + 주문 상품 저장 (CASCADE ALL)
    Order order = Order.builder()
        .user(user).totalPrice(totalPrice)
        .shippingAddress(shippingAddress).build();
    for (Product product : products) {
        order.addItem(OrderItem.builder()
            .product(product).quantity(qty)
            .purchasePrice(product.getPrice()) // 주문 시점 가격 스냅샷
            .build());
    }
    return orderRepository.save(order);
}
부하 테스트 관점: 여러 가상 사용자가 동시에 같은 상품에 주문하면 decreaseStock()에서 DB row-level lock 경합이 발생합니다. 이 경합 상황에서의 응답 시간과 에러율이 핵심 측정 포인트입니다.

📚 Gatling 핵심 개념

Simulation 클래스 구조

모든 Gatling 시뮬레이션은 Simulation을 상속하고 세 가지 요소를 정의합니다:

public class MySimulation extends Simulation {

    // 1. HTTP 프로토콜 설정 (공통 baseUrl, 헤더)
    HttpProtocolBuilder httpProtocol = http
        .baseUrl("http://localhost:8080")
        .contentTypeHeader("application/json")
        .acceptHeader("application/json")
        .shareConnections();  // TCP 커넥션 풀 공유

    // 2. 시나리오 정의 (가상 사용자의 행동 시퀀스)
    ScenarioBuilder scn = scenario("시나리오 이름")
        .exec(http("요청 이름").get("/api/...")
            .check(status().is(200)));

    // 3. 부하 주입 + 성능 기준 (인스턴스 초기화 블록)
    {
        setUp(
            scn.injectOpen(rampUsers(50).during(Duration.ofSeconds(30)))
        )
        .protocols(httpProtocol)
        .assertions(global().responseTime().percentile(95.0).lt(1000));
    }
}

Session (세션)

가상 사용자 1명에 귀속된 불변 Map 저장소. 요청 간 데이터를 전달합니다.

// 값 저장
.exec(session -> session
    .set("userId", 42))

// JSON 응답에서 추출 → 저장
.check(jsonPath("$.id")
    .saveAs("userId"))

// 이후 요청에서 참조
.get("/api/users/#{userId}")

Check (검증)

응답의 상태 코드, JSON 필드, 헤더 등을 검증합니다. 실패 시 요청은 KO로 기록됩니다.

// 상태 코드 검증
.check(status().is(201))

// 값 검증
.check(jsonPath("$.status")
    .is("PENDING"))

// 존재 여부만 확인
.check(jsonPath("$.totalElements")
    .exists())

// 에러를 성공으로 처리
.check(status().is(400)) // 400이 정상

Pause (생각 시간)

실제 사용자는 요청 사이에 페이지를 읽거나 폼을 작성합니다. pause()로 이 "생각 시간"을 시뮬레이션합니다.

// 200ms ~ 500ms 사이 랜덤 대기
.pause(Duration.ofMillis(200), Duration.ofMillis(500))

// pause 없이 연속 요청 → 비현실적 부하 (서버에 과도한 부담)
// pause 너무 길면 → 실제보다 적은 부하 (의미 없는 테스트)

📈 부하 주입 모델 (Injection Models)

Open Model vs Closed Model

Open Model (이 프로젝트에서 사용)

진행 중인 사용자 수와 무관하게 새 사용자를 계속 투입합니다.

  • 실제 웹 서비스 트래픽 패턴과 유사
  • 서버가 느려져도 부하는 계속 증가
  • injectOpen()으로 설정

Closed Model

동시 접속자 수를 일정하게 유지합니다.

  • 한 명이 완료해야 다음 한 명 투입
  • 커넥션 풀 테스트에 적합
  • injectClosed()로 설정

이 프로젝트에서 사용된 주입 패턴

패턴문법용도사용 시뮬레이션
nothingFor nothingFor(2s) JVM 워밍업 대기, 시나리오 간 시차 조절 전체
rampUsers rampUsers(50).during(30s) 점진적 부하 증가 (출근 시간대 패턴) User, Order, Post
constantUsersPerSec constantUsersPerSec(5).during(30s) 일정 QPS에서의 안정성 측정 (Steady State) User, Order, Post
atOnceUsers atOnceUsers(10) 스파이크 테스트 (이벤트/프로모션 시작) Order, Post
rampUsersPerSec rampUsersPerSec(1).to(5).during(20s) 주입 속도 자체를 점진적으로 증가 Post (reader)
조합 전략: 실제 프로젝트에서는 이 패턴들을 조합합니다. 예를 들어 PostCommentSimulation의 readerFlow는 atOnceUsers(20)으로 초기 스파이크를 만들고, 이어서 rampUsersPerSec(1).to(5)로 꾸준히 증가시킵니다 — "처음에 몰리고 이후에도 꾸준히 유입되는" 실제 이벤트 트래픽을 재현합니다.

✅ Assertion & 성능 기준

Assertion 문법

// 전체 요청에 대한 기준
global().responseTime().percentile(95.0).lt(1000)  // p95 < 1초
global().successfulRequests().percent().gte(95.0)    // 성공률 ≥ 95%

// 개별 요청별 기준 (forAll)
forAll().responseTime().mean().lt(500)               // 모든 요청의 평균 < 500ms

// 비교 연산자
.lt(N)   // less than (미만)
.lte(N)  // less than or equal (이하)
.gt(N)   // greater than (초과)
.gte(N)  // greater than or equal (이상)

시뮬레이션별 성능 기준 비교

시뮬레이션p95 응답시간성공률기준 설정 이유
User < 1,000ms ≥ 95% 단순 CRUD + 검색 쿼리만 포함
Order < 2,000ms ≥ 80% 재고 차감 트랜잭션 + 에러 시나리오 포함
Post < 1,500ms ≥ 95% 읽기/쓰기 혼합 + viewCount UPDATE 포함
왜 평균(mean)이 아닌 p95를 사용하는가?
평균 응답시간은 이상값(outlier)에 의해 왜곡됩니다. p95는 "100명 중 95명은 이 시간 안에 응답받는다"는 뜻으로, 실제 사용자 경험을 더 정확히 반영합니다. p99는 더 엄격한 기준으로, SLO(Service Level Objective) 정의에 자주 사용됩니다.

💡 발견할 수 있는 인사이트

1. DB 병목 지점 식별

  • 재고 차감 Lock 경합: 동일 상품에 동시 주문 시 decreaseStock()에서 row-level lock 대기 발생. p95 응답 시간이 급격히 증가하면 비관적/낙관적 락 전략 변경 필요
  • viewCount UPDATE 경합: 게시글 조회 시 매번 UPDATE 실행. 읽기 부하가 높아지면 DB write lock이 읽기 성능까지 저하시킬 수 있음
  • N+1 쿼리 문제: Order 조회 시 Fetch Join 미사용 시 OrderItem 수만큼 추가 쿼리 발생. 응답 시간이 데이터 수에 비례하여 증가

2. 쿼리 성능 분석

  • QueryDSL 동적 쿼리: 조건 조합(카테고리+가격+재고)에 따른 실행 계획 변화. 인덱스 유무에 따라 성능 차이가 크게 남
  • 정렬 성능: sort=viewCount,desc와 sort=createdAt,desc의 응답 시간 비교. 인덱스가 없는 컬럼 정렬은 Full Table Scan 유발
  • 페이징 깊이: page=0은 빠르지만 page=100은 느려질 수 있음 (Offset 방식의 한계)

3. 서버 동작 패턴 분석

  • JVM 워밍업 효과: 첫 수 초간의 응답 시간과 안정화 후의 응답 시간 비교. JIT 컴파일 완료 후 성능이 개선되는지 확인
  • 에러 처리 비용: 정상 응답(200/201)과 에러 응답(400/404)의 응답 시간 비교. Exception 생성 비용, 스택 트레이스 비용 분석
  • Steady State 안정성: constantUsersPerSec 구간에서 응답 시간이 일정한지, 서서히 증가하는지 (메모리 누수, 커넥션 풀 고갈 가능성)

4. 실서비스 적용을 위한 개선 포인트
